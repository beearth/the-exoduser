## 3. 크롬 성능 병목 — 원인 확정

### 병목 구조 (코드 분석 결과)

```
적 1마리 렌더 사이클:

1. whiteTex 세팅 → 글로우 arc fill       [_setTex → _flush]
2. spriteTexA 세팅 → 본체 drawImage     [_setTex → _flush]  
3. whiteTex 세팅 → 마커 fillRect         [_setTex → _flush]
4. spriteTexB 세팅 → 다음 적 본체        [_setTex → _flush]
5. whiteTex 세팅 → 다음 적 글로우        [_setTex → _flush]
...반복...
```

**적 100마리 = 텍스처 스위칭 200~300회/프레임 = draw call 300+**

### 왜 파이어폭스는 빠른가

```
Firefox: WebGL2 → 네이티브 OpenGL → 텍스처 바인드 비용 낮음
Chrome:  WebGL2 → ANGLE → DirectX 변환 → 텍스처 바인드 비용 높음
                  ↑
            이 변환 레이어가 텍스처 스위칭마다 오버헤드 발생
```

### _getESpr 시스템의 문제 (line 1920~1932)

```javascript
// 현재: 색상+속성+크기 조합마다 별개 캔버스 생성
function _getESpr(col, el, r){
  const k = col+'_'+el+'_'+r;  // 조합 키
  let c = _eSpCache.get(k);    // 캐시 히트?
  if(c) return c;
  // ... 새 캔버스 생성 (arc+stroke 그리기)
  _eSpCache.set(k, c);         // 캐시에 저장
  return c;                     // 개별 캔버스 = 개별 텍스처!
}
```

캐시 100개 제한 + 초과 시 전체 clear → 텍스처 전부 재생성 → 추가 성능 스파이크

## 4. 성능 Tier 시스템 (수정)

### Tier 재정의 (아틀라스 도입 후 기준)

v1.0 Tier는 "아틀라스 없이" 기준이었음. 아틀라스 도입 후 기준 상향:

| Tier | 조건 | 해상도 | 파티클 | 후처리 | 최대 몹 | AI 주기 |
|------|------|--------|--------|--------|---------|--------|
| **S** | 아틀라스+GPU, 초기 avg 120fps+ | 1.0 | 150% | 풀 ON | 400+ | 60Hz |
| **A** | 아틀라스+GPU, 초기 avg 60~120fps | 1.0 | 100% | 블룸/트레일 ON | 300 | 60Hz |
| **B** | 아틀라스+GPU, 초기 avg 40~60fps | 0.85 | 70% | 트레일만 | 200 | 30Hz |
| **C** | WARP/SW렌더, fps 40 미만 | 0.65 | 25% | 전부 OFF | 150 | 15Hz |

### 기존 v1.0 대비 변경점

```
v1.0 Tier A: 400+ 몹, 블룸 ON
v2.0 Tier S: 400+ 몹, 풀 이펙트 (아틀라스 덕에 가능)
v2.0 Tier A: 300 몹, 대부분 이펙트 ON (이전 Tier A 수준)

차이: 아틀라스 도입으로 Tier A가 사실상 이전 Tier S급 성능
```

### 5-2. 코드 변경

```javascript
// ═══ 변경 전 (line 1920~1932) ═══
const _eSpCache = new Map();
function _getESpr(col, el, r){
  const k = col+'_'+el+'_'+r;
  let c = _eSpCache.get(k); if(c) return c;
  if(_eSpCache.size >= 100) _eSpCache.clear(); // 100개 초과 시 전체 초기화!
  c = document.createElement('canvas');
  // ... arc + stroke 원형 그리기
  _eSpCache.set(k, c); return c;
}

// ═══ 변경 후 ═══
const _atlas = new Image(); // 1장 아틀라스
const _atlasUV = {};        // etype → [{u0,v0,u1,v1}, ...] 프레임별 UV
let _atlasReady = false;

_atlas.src = 'img/enemy_atlas.png';
_atlas.onload = () => {
  _atlasReady = true;
  // UV 좌표 테이블 빌드 (레이아웃 기반)
  buildAtlasUV();
};

function drawEnemy(e, frame){
  if(!_atlasReady) return drawEnemyCircle(e); // 폴백
  const uv = _atlasUV[e.etype][frame];
  X.drawImage(_atlas,
    uv.sx, uv.sy, uv.sw, uv.sh, // 소스 (아틀라스 내 좌표)
    e.x - uv.sw/2, e.y - uv.sh/2, uv.sw, uv.sh // 목적지
  );
  // 같은 _atlas 텍스처 → 배칭 유지 → flush 안 일어남!
}
```

### 5-3. 속성 틴팅

```
방법: 초기 1회 속성별 아틀라스 생성 (오프스크린)

원본 아틀라스 1장 (1024×1024)
  → 화염 틴트 아틀라스 (주황 오버레이)
  → 빙결 틴트 아틀라스 (파랑 오버레이)
  → 뇌전 틴트 아틀라스 (노랑 오버레이)
  → 암흑 틴트 아틀라스 (보라 오버레이)
  → 물리 = 원본 그대로

총 5장 × 1024×1024 = 약 5MB VRAM
텍스처 스위칭: 속성 변경 시만 (적 타입별 정렬하면 최소화)
```

## 11. 출시 플랫폼 전략

### 11-1. 플랫폼별 분석

| 플랫폼 | 장점 | 단점 | 성능 |
|--------|------|------|------|
| **브라우저 (itch.io)** | 즉시 출시, 진입장벽 0 | 크롬 성능 이슈 | 아틀라스로 해결 |
| **Steam (Electron)** | 수익화, 스토어 노출 | Chromium 기반=크롬 성능 | 아틀라스 필수 |
| **Steam (NW.js)** | Electron보다 가벼움 | 같은 Chromium 기반 | 아틀라스 필수 |
| **Unity 이식** | 네이티브 성능, 풀 기능 | 이식 2~3개월 | 성능 제약 없음 |

### 11-2. 추천 경로

```
[Phase A] 아틀라스 최적화 + 스프라이트 교체 (2주)
    ↓
[Phase B] itch.io 무료 데모 출시 (즉시)
    ↓ 반응 수집
[Phase C] Steam Electron 래핑 얼리억세스 (1주)
    ↓ 수익 발생
[Phase D] Unity 이식 정식 출시 (2~3개월)
```

### 11-3. Steam HTML5 주의사항

```
해결되는 것:
  ✅ 브라우저 호환 문제 (Chromium 고정 버전)
  ✅ "어떤 브라우저 써야 하나요?" 질문 제거
  ✅ URL 노출 없음 (코드 보호)

해결 안 되는 것:
  ❌ 크롬 ANGLE 성능 문제 그대로 (아틀라스로 해결 필수)
  ❌ Steamworks SDK 연동 (업적/클라우드세이브 추가 작업)
  ❌ "네이티브 게임" 기대치 (리뷰에서 "최적화 구림" 위험)
  ❌ Electron 자체 메모리 오버헤드 (~100MB)
```

### 11-4. Vampire Survivors 선례

```
동일 경로:
  HTML5 프로토 → itch.io → Steam 얼리억세스 → 네이티브 이식

교훈:
  - HTML5로도 Steam 대박 가능 (초기 EA 기간)
  - 단, 인기 터진 후 네이티브 이식은 필수였음
  - "시스템 깊이 > 그래픽 품질" 전략이 먹힘
```

## 16. 성능 예산

### 프레임당 허용 (크롬 60fps = 16.6ms)

| 항목 | 현재 (ms) | 아틀라스 후 (ms) | +이펙트 후 (ms) |
|------|----------|----------------|---------------|
| 로직 (update) | 4~8 | 4~8 | 4~8 |
| 맵 렌더 | 1~2 | 1~2 | 1~2 |
| 적 렌더 (150) | 8~14 | **1~3** | 2~4 |
| 투사체 (50) | 1~2 | 1~2 | 1~2 |
| 파티클 (300) | 1~2 | 1~2 | 1~3 |
| 포스트프로세싱 | 0 | 0 | 0.5~1.5 |
| UI/텍스트 | 1~2 | 1~2 | 1~2 |
| **합계** | **16~30** | **9~19** | **11~22** |
| **크롬 FPS** | **33~62** | **52~111** | **45~90** |

### 핵심 인사이트

```
아틀라스만 넣으면:
  적 렌더 비용 80% 절감 → 크롬에서 여유 7~11ms 확보
  
확보된 여유로:
  이펙트+포스트프로세싱 추가 가능 (1.5~3ms 소비)
  
최종:
  크롬 60fps 안정 + 비주얼 대폭 강화 = 동시 달성
```
